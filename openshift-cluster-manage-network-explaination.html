⚠ Throughput Insight

Ingress VIP is active/passive via keepalived. Only 1 of 3 router pods receives traffic at any time. Effective app throughput: maxconn 50,000 (single pod), not 150,000. Other 2 router pods are hot standby. API traffic is different — KNI HAProxy on VIP holder round-robins across all 3 apiservers.

Why nftables DNAT? Why not use port 6443 directly?
▶
The Problem

kube-apiserver binds on *:6443 (all interfaces). The API VIP 192.168.37.201 lives on br-ex on the VIP holder node. Without nftables, traffic to VIP:6443 hits the local apiserver directly — only 1 of 3.

✗ Without DNAT
Client → VIP:6443
↓
kube-apiserver (1 node only)
  • No load balancing — 1 apiserver handles everything
  • No health checking — degraded apiserver still gets traffic
  • Other 2 apiservers sit idle
✓ With DNAT (current setup)
Client → VIP:6443
↓
nftables: 6443 → 9445
↓
HAProxy → 3 apiservers
  • Load balanced — roundrobin across all 3
  • Health-aware — checks /readyz every 5s
  • Transparent — clients still use standard port 6443
Two Failure Domains
keepalivedDetects node-level failure → moves VIP to another node
HAProxyDetects apiserver-level failure → stops sending traffic to unhealthy backend

A node can be healthy (keepalived happy) while its apiserver is degraded (accepting connections but returning errors). HAProxy catches this; direct 6443 does not.

The nftables Rules
# External traffic (arriving from network)
chain prerouting {
  type nat hook prerouting priority dstnat;
  ip daddr 192.168.37.201 tcp dport 6443
    dnat to 192.168.37.201:9445
    # counter: 14,936 packets / 898,175 bytes
}

# Local traffic (kubelet, controllers on same node)
chain output {
  type nat hook output priority dstnat;
  ip daddr 192.168.37.201 oifname "lo" tcp dport 6443
    redirect to :9445
    # counter: 2,899 packets / 173,940 bytes
}
Why no HAProxy in front of Ingress Router?
▶
The Router IS Already HAProxy

Putting HAProxy in front of HAProxy = double proxy. Every app request would traverse two proxy hops, adding latency to all user-facing traffic. For API this is acceptable (low volume, latency-tolerant control plane). For app traffic serving thousands of requests/sec, it hurts.

Why API Gets HAProxy But Ingress Doesn't
Traffic typeAPI = control plane (low volume) vs Ingress = data plane (high volume)
Backendkube-apiserver has no built-in LB. Router IS an LB — adding another is redundant
Double-proxy costAPI: acceptable (infrequent calls). Apps: every HTTP request pays the penalty
LB valueAPI: real gain (3 apiservers share load). Ingress: marginal (just picks which router does L7)
The Right Fix for Ingress HA on Bare Metal

Active/passive VIP is a known bare-metal limitation. Correct solutions eliminate proxy overhead entirely:

MetalLB + BGP/ECMPAdvertises VIP from multiple nodes. Upstream router distributes at L3. All 3 routers active. No extra proxy.
External LBF5, HAProxy, or nginx in front. Distributes to all router nodes on 80/443 directly.
Sharded IngressMultiple IngressControllers with separate VIPs. Split traffic by domain.
