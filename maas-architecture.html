02 — On the Wire

Step-by-Step Request Journey

What actually happens, hop by hop, when a client sends POST /swongpai-maas/redhataillama-31-8b-instruct/v1/chat/completions. Every step below is verified against the live cluster configuration.

1
Client → AWS NLB
HTTPS 443
Client resolves maas.apps.rosa.rosa-h100-jp.g4b0.p3.openshiftapps.com (Route53 → NLB a83033…elb.amazonaws.com) and sends the request with header Authorization: Bearer sk-oai-…. The NLB forwards raw TCP to the gateway pod — no TLS termination here.
2
NLB → Envoy Gateway
TLS terminate
Pod maas-default-gateway-openshift-default (an istio-proxy Envoy managed by istiod-openshift-gateway) terminates TLS using the maas-gateway-tls secret (ROSA wildcard cert). From here on, Kuadrant's WASM filter chain runs inside this Envoy for every request.
3
Envoy → Authorino
gRPC ext_authz :50051
Envoy pauses the request and sends a CheckRequest to Authorino. The AuthPolicy matches the key pattern ^Bearer sk-oai-.* and validates the key by calling maas-api:8443/internal/v1/api-keys/validate (which checks PostgreSQL); a second metadata call to /internal/v1/subscriptions/select resolves the user's subscription. Two OPA/Rego policies then run: auth-valid (is the credential valid?) and require-group-membership (is the user in cluster-admins or rhods-admins?). Results are cached 60s. Failure → 401 / 403. OpenShift tokens are also accepted — but only for the /v1/models listing endpoint.
4
Authorino → Envoy
identity metadata
On success Authorino returns OK plus identity metadata: userid, selected_subscription (e.g. models-as-a-service/admin-sub@swongpai-maas/redhataillama-31-8b-instruct), and group list. Envoy stores these as dynamic metadata — the rate limiter and telemetry both read them downstream.
5
Envoy → Limitador
gRPC ratelimit :8081
Envoy asks Limitador: does user auth.identity.userid still have token budget? The live TokenRateLimitPolicy allows 1000 tokens per 1-minute window per user for this subscription (the /v1/models path is exempt). A gateway-wide deny-all-by-default policy (limit 0) blocks any path that isn't a configured model or the maas-api. Over budget → 429.
6
Envoy: HTTPRoute match
URLRewrite
The HTTPRoute has separate rules for /v1/completions, /v1/chat/completions, and /v1/responses — each strips the /swongpai-maas/redhataillama-31-8b-instruct prefix and forwards to the InferencePool backend (a Gateway API extension, not a normal Service). A catch-all rule routes anything else directly to the workload Service.
7
Envoy → EPP
gRPC ext_proc :9002
Because the backend is an InferencePool, Envoy consults the Endpoint Picker (EPP) before choosing a pod. The live EPP config runs three plugins: queue-scorer (weight 2 — prefer pods with shorter queues), prefix-cache-scorer (weight 3 — prefer the pod that already has this prompt's prefix in KV cache), and max-score-picker (choose the highest combined score). The EPP's tokenizer sidecar tokenizes the prompt to compute prefix-cache matches.
8
Envoy → vLLM pod
HTTP :8000
The rewritten request (POST /v1/chat/completions) lands on the chosen vLLM pod. vLLM runs the model on its MIG 3g.40gb slice (40GB VRAM), reusing cached KV tensors when the prefix matched. Weights were loaded at startup from the OCI modelcar image modelcar-llama-3-1-8b-instruct:1.5 — no external download, no PVC.
9
Response → payload-processing
gRPC ext_proc :9004
As the response streams back through Envoy, the payload-processing service inspects the response body and extracts usage.total_tokens from the OpenAI-format JSON. This count is reported to Limitador, which decrements the user's remaining token budget — this is how token-based (not request-based) rate limiting works.
10
Envoy → Client
HTTPS 200
The completed response returns to the client. In parallel, the TelemetryPolicy enriches Envoy's Prometheus metrics with model, user, subscription, organization_id, and cost_center labels — feeding the MaaS observability dashboard for usage attribution and showback.

06 — The Brain of the Data Plane

llm-d & the Endpoint Picker

llm-d is a Kubernetes-native distributed inference framework (Red Hat, Google, CoreWeave, IBM, NVIDIA — built on vLLM and the Gateway API Inference Extension). RHOAI 3.4 ships it as the engine behind every LLMInferenceService. Its core idea: the load balancer should understand LLMs.

Why Classic Load Balancing Fails for LLMs

Round-robin and least-connections assume all requests cost roughly the same and any backend can serve them equally well. Both assumptions break for LLMs:

  1. Requests are wildly unequal. A 10-token prompt and a 10,000-token prompt differ by ~1000× in prefill compute. "Number of connections" tells you nothing about actual GPU load — one pod may hold 3 trivial requests, another 1 giant one.
  2. Backends are not interchangeable — KV cache makes them stateful. When a vLLM pod processes a prompt, it stores the computed attention keys/values (KV cache) in GPU memory. A follow-up request sharing that prefix (same system prompt, same chat history) can skip recomputation only if it lands on the same pod. Blind balancing throws that away and repays full prefill cost on every hop.
  3. Generation length is unknown at admission. A request that generates 4,000 tokens occupies KV memory for its whole lifetime. Queue depth right now is a weak signal for load 30 seconds from now — you want scheduling that reads engine-level signals, not TCP-level ones.

llm-d's answer: pull the routing decision out of the proxy and give it to a model-aware sidecar service — the Endpoint Picker (EPP). Envoy still moves the bytes; the EPP decides where.

Worked example — a user sends the 3rd turn of a chat session; the 2,400-token conversation prefix is cached on Pod 1, which is currently busier (illustrative scores, 0–1 normalized):

Podqueue-scorer (×2)prefix-cache-scorer (×3)Total
Pod 1 — busy, cache warm0.20 × 2 = 0.400.95 × 3 = 2.853.25
Pod 2 — idle, cache cold0.90 × 2 = 1.800.00 × 3 = 0.001.80

The EPP sends the request to the busier pod — deliberately. Reusing 2,400 cached tokens skips the entire prefill for that prefix, cutting time-to-first-token far more than a shorter queue would. That trade-off is exactly what the 3-vs-2 weighting encodes. If no pod has a cache hit, prefix scores are all zero and queue depth decides — graceful fallback to load-based balancing.

Full request lifecycle through the EPP:

  1. Envoy matches the HTTPRoute to the InferencePool and opens an ext_proc conversation with the EPP, streaming request headers + body.
  2. The tokenizer sidecar converts the prompt to token IDs — the same tokenization the model itself will use (its modelcar container symlinks the model files at /mnt/models).
  3. The prefix-cache index is consulted: longest-prefix match of those token IDs against each pod's known KV blocks (index kept current by ZMQ KV events, not polling).
  4. queue-scorer reads current queue depth per pod; both scorers emit 0–1 scores.
  5. max-score-picker computes Σ(score × weight) and picks the winner.
  6. EPP returns the chosen pod's address in a routing header; Envoy forwards the request to that exact pod — not the Service VIP.
  7. The pod runs inference; new KV blocks it creates are announced over ZMQ, updating the index — so the next turn of this conversation scores a hit and sticks to the same pod.

If the EPP itself dies: the InferencePool is configured failureMode: FailOpen — Envoy falls back to cache-blind balancing across the pool. Latency degrades; traffic survives.

Beyond This Cluster — the llm-d Ladder

llm-d defines three "well-lit paths" — proven deployment profiles in increasing order of scale. This cluster runs the first. Evidence the higher rungs are already wired in: the LLMInferenceService status carries preset annotations for config-llm-prefill-template and config-llm-decode-worker-data-parallel — unused at replicas: 2, but one spec change away.

this cluster

1 · Intelligent Inference Scheduling

Identical replicas + EPP with queue & prefix-cache scorers. Lower TTFT and higher goodput than any cache-blind balancer, with no change to the model pods themselves. Fits single-node and modest multi-GPU setups.

next rung

2 · Prefill / Decode Disaggregation

Split the two phases onto different pods: prefill is compute-bound (long prompts), decode is memory-bandwidth-bound (token streaming). Specialized pods each run at their bottleneck; computed KV tensors are transferred prefill→decode over a fast interconnect. Pays off for long-context, high-QPS serving.

large scale

3 · Wide Expert Parallelism

For giant Mixture-of-Experts models (DeepSeek-R1 class): experts sharded across many GPUs/nodes with data-parallel attention, deployed as multi-node leader/worker groups. The EPP's role grows to routing across heterogeneous multi-node serving groups.

07 — Deep Dive

How It Works

The mechanics behind the diagrams — with the actual configuration from the live cluster, collapsible below each explanation.

Two Chains of Command

The intent chain (humans → policy): An admin writes three small CRs — MaaSSubscription (who gets how many tokens), MaaSAuthPolicy (who may access which models), MaaSModelRef (which model). The maas-controller watches these and generates the low-level enforcement objects: a Kuadrant AuthPolicy with OPA rules and a TokenRateLimitPolicy with counters, both attached to the model's HTTPRoute.

The serving chain (model → pods): A user creates one LLMInferenceService. The llmisvc-controller expands it into the entire serving stack: vLLM Deployment (2 replicas), EPP scheduler Deployment, InferencePool, HTTPRoute with URL rewrites, and a MaaSModelRef that self-registers the model with MaaS.

Neither chain touches the other's objects — they meet only at the HTTPRoute, where the policy attaches to the route the serving chain created.

Deny by Default

The gateway ships with a zero-trust posture enforced by two gateway-wide policies:

  1. gateway-default-auth — an AuthPolicy that denies requests to any model that has no explicit MaaSAuthPolicy. A model-specific policy overrides this default for its route.
  2. gateway-default-deny — a TokenRateLimitPolicy with limit: 0 for any path that isn't /maas-api or /v1/models. No subscription = no tokens = 429 on the first request.
  3. maas-default-gateway-authn — accepts OpenShift service-account tokens via TokenReview + SubjectAccessReview against serving.kserve.io resources, extracting the target namespace and model from the URL path. This is how the RHOAI dashboard itself can browse models without an API key.

Adding a model therefore opens exactly one path through the gate — everything else stays shut.

Routing — One Route, Four Rules

The HTTPRoute created by the llmisvc-controller exposes the model under a namespaced path prefix and rewrites it to the OpenAI paths vLLM expects. Three explicit API rules go to the InferencePool (so the EPP picks the pod); a catch-all rule (health checks, tokenizer info, metrics) bypasses the EPP and goes straight to the workload Service.

Authentication & Authorization — the Full Ruleset

The model's AuthPolicy accepts three credential types, evaluated in priority order: API keys matching ^Bearer sk-oai-.* (priority 0), then OpenShift/OIDC tokens (priority 1, only for /v1/models listing). Authorization is two OPA/Rego checks — credential validity, then group membership — each cached for 60 seconds to keep per-request latency negligible.

AuthPolicy — authentication + authorization (abridged)live cluster 
rules:
  authentication:
    api-keys:
      plain: {selector: request.headers.authorization}
      priority: 0
      when:
      - {selector: request.headers.authorization, operator: matches, value: ^Bearer sk-oai-.*}
    kubernetes-tokens:
      kubernetesTokenReview:
        audiences: [https://oidc.op1.openshiftapps.com/…]
      priority: 1
      when:
      - {selector: request.url_path, operator: matches, value: .*/v1/models$}
  authorization:
    auth-valid:
      cache: {ttl: 60}
      opa:
        rego: |
          allow { input.auth.metadata.apiKeyValidation.valid == true }
          allow { input.auth.identity.user.username != "" }
          allow { input.auth.identity.sub != "" }
    require-group-membership:
      cache: {ttl: 60}
      opa:
        rego: |
          allowed_groups := ["cluster-admins","rhods-admins"]
          # … username extraction from key / OIDC / k8s token …
          # allow when user's groups intersect allowed_groups

GPU Layout — NVIDIA H100 MIG

The H100 80GB is partitioned with Multi-Instance GPU (MIG) into two isolated 3g.40gb slices — separate compute, memory, and L2 cache, zero noisy-neighbor interference. With mig.strategy=single both slices surface as the plain nvidia.com/gpu resource, so each vLLM pod requests nvidia.com/gpu: 1 and lands on its own slice. Both replicas of the model run on the same physical GPU — yet fully isolated.

Why 2 replicas beat 1 big one here: the EPP spreads sessions across both slices, prefix-cache routing keeps repeat users sticky to "their" cache, and one slice can drain/restart while the other keeps serving.
